# Overview

Fragola Agents expose lifecycle events so you can:
- Inspect and mutate user and assistant messages.
- Intercept tool calls before and after execution.
- Adjust model invocation config or transform streamed chunks.
- Observe step completion and state transitions for persistence, metrics, or orchestration.

You register handlers on the `Agent` itself (not on `AgentContext`) using methods like:
- `agent.onUserMessage(...)`
- `agent.onAiMessage(...)`
- `agent.onBeforeStep(...)`
- `agent.onAfterStep(...)`
- `agent.onBeforeModelInvocation(...)`
- `agent.onAfterModelInvocation(...)`
- `agent.onBeforeToolCall(...)`
- `agent.onToolCall(...)`
- `agent.onModelInvocation(...)`
- `agent.onAfterToolCall(...)`
- `agent.onAfterStateUpdate(...)`

Each method:
- Returns an **unsubscribe** function.
- Receives an `AgentContext` instance so handlers can read state, stores, tools, and instructions.
- Runs in registration order.

Handlers that transform a value can return:
- the replacement value
- `skip()` to keep the current value unchanged and continue to the next handler
- `context.stop()` or `context.stopSync()` to abort the current pipeline early

Observation-only events such as `after:*` ignore replacement values and are best used for side-effects.

<Callout type="warn">
  Event handlers run inside the Agent execution loop. Avoid blocking operations and prefer short, deterministic logic or well‑bounded async calls.
</Callout>

# onUserMessage
Register a handler that runs when a user message is appended. Handlers can **inspect or mutate** the message before it is stored.

Use this to:
- Normalize/clean user input.
- Attach metadata (routing hints, user IDs, feature flags).
- Enforce guardrails (e.g. strip unsupported content types).

**Examples**
```ts
// 1) Trim whitespace and collapse multiple spaces
agent.onUserMessage((message) => {
  const content = typeof message.content === 'string'
    ? message.content.replace(/\s+/g, ' ').trim()
    : message.content;
  return { ...message, content };
});

// 2) Attach tenant metadata used by tools
agent.onUserMessage((message, context) => {
  return {
    ...message,
    meta: {
      ...(message as any).meta,
      tenantId: context.instance.store?.value.activeTenantId,
    },
  } as any;
});

// 3) Block empty messages
agent.onUserMessage((message) => {
  if (typeof message.content === 'string' && !message.content.trim()) {
    throw new Error('Empty messages are not allowed');
  }
  return message;
});
```

# onAiMessage
Register a handler that runs when an assistant message is generated or streamed. Handlers can **inspect and replace** the AI message.

The callback signature is:
- `(message, finish_reason, usage, context) => maybePromise<message>`

`finish_reason === null` means the stream is still in progress.

Use this to:
- Post‑process final answers (redaction, formatting, translation).
- Inspect streaming partials for live UIs.
- Enforce style guides or add disclaimers.

**Examples**
```ts
// 1) Append a disclaimer to final answers only
agent.onAiMessage((message, finish_reason) => {
  if (finish_reason === null) return message;
  if (typeof message.content === 'string') {
    return {
      ...message,
      content: message.content + '\n\n_Disclaimer: experimental output._',
    };
  }
  return message;
});

// 2) Redact secrets from any assistant content
agent.onAiMessage((message) => {
  if (typeof message.content === 'string') {
    return {
      ...message,
      content: message.content.replace(/sk-[a-zA-Z0-9]+/g, '[REDACTED_KEY]'),
    };
  }
  return message;
});

// 3) Mirror assistant messages into a metrics store
agent.onAiMessage((message, finish_reason, usage, context) => {
  if (finish_reason === null) return message;
  const metrics = context.getStore('metrics');
  metrics?.update(v => ({
    ...v,
    lastAiMessageLength: (message.content as string)?.length ?? 0,
  }));
  return message;
});
```

# onBeforeStep
Register a handler that runs before a step starts.

Use this to:
- clamp per-call step options
- flip feature flags for the next step
- stop execution before the step begins

**Example**
```ts
agent.onBeforeStep((options) => {
  if ((options.maxStep ?? 10) <= 2) return options;
  return { ...options, maxStep: 2 };
});
```

---

# onAfterStep
Register a handler that runs after a step completes.

The callback receives the resolved options, the messages appended during the step, and how many
model generations happened in that step.

**Example**
```ts
agent.onAfterStep((options, newMessages, stepsTaken, context) => {
  console.log('step completed', {
    stepsTaken,
    appendedMessages: newMessages.length,
    status: context.state.status,
  });
});
```

# onBeforeModelInvocation
Register a handler that runs before the model request is made.

The callback receives the current config and may:
- return `{ modelSettings, clientOptions }` to override the request
- return `{ injectMessage }` to bypass the API call with a final assistant message
- return `{ injectResponse }` to provide a custom SDK response
- return `skip()` to preserve the current config
- return `context.stop()` to cancel the invocation

**Examples**
```ts
// 1) Override request settings for the next invocation
agent.onBeforeModelInvocation((config) => {
  if (!("modelSettings" in config)) return config;
  return {
    modelSettings: {
      ...config.modelSettings,
      temperature: 0,
    },
    clientOptions: config.clientOptions,
  };
});

// 2) Inject a synthetic assistant reply without calling the API
agent.onBeforeModelInvocation(() => ({
  injectMessage: { content: 'hello from cache' },
}));
```

# onModelInvocation
Register a handler that can inspect or transform model invocation payloads.

Streamed responses emit `kind === 'chunk'` before each chunk is merged into the partial assistant
message. Returning a replacement chunk changes what later handlers and `onAiMessage` receive.

**Examples**
```ts
// 1) Remove debug markers from streamed content
agent.onModelInvocation((kind, data) => {
  if (kind !== 'chunk') return data;
  const choice = data.choices?.[0];
  const content = choice?.delta?.content;
  if (!choice || !content) return data;

  return {
    ...data,
    choices: [{
      ...choice,
      delta: {
        ...choice.delta,
        content: content.replace('[DEBUG]', ''),
      },
    }],
  };
});

// 2) Abort a stream after the first chunk that matches a condition
agent.onModelInvocation((kind, data, context) => {
  if (kind !== 'chunk') return data;
  const content = data.choices?.[0]?.delta?.content;
  if (content?.includes('[ABORT]')) {
    return context.stop();
  }
  return data;
});
```

# onAfterModelInvocation
Register a handler that runs once the final assistant message has been resolved.

Use this to:
- inspect the final assistant message after injection or model completion
- record metrics for completed model turns
- trigger side-effects that should only run once per invocation

**Example**
```ts
agent.onAfterModelInvocation((message, context) => {
  console.log('assistant turn complete', {
    hasToolCalls: Boolean(message.tool_calls?.length),
    status: context.state.status,
  });
});
```

# onBeforeToolCall
Register a handler that runs after tool arguments are parsed and before the tool handler executes.

The callback receives `{ params }` initially and may:
- return `{ params }` to rewrite validated arguments
- return `{ injectResult }` to bypass the handler entirely
- return `skip()` to preserve the current config
- return `context.stop()` to cancel the current tool call

**Examples**
```ts
// 1) Rewrite validated params before the handler runs
agent.onBeforeToolCall((config, tool) => {
  if (tool.name !== 'search' || !("params" in config)) return config;
  return {
    params: {
      ...config.params,
      limit: 5,
    },
  };
});

// 2) Provide a result for a dynamic tool
agent.onBeforeToolCall(async (config, tool) => {
  if (tool.name !== 'dynamic_action' || !("params" in config)) return config;
  return {
    injectResult: await performDynamicAction(config.params),
  };
});
```

# onToolCall
Register a handler that runs once a tool result exists.

This event receives the current result, the effective params, and the tool reference. Each handler
may transform the result before it is passed to later `toolCall` handlers, `onAfterToolCall`, and
the `role: 'tool'` message appended to state.

Use this to:
- normalize handler output
- attach derived metadata to the final tool result
- log or audit tool execution without re-running the handler

**Examples**
```ts
// 1) Wrap the final tool result with metadata
agent.onToolCall((result, params, tool) => {
  if (tool.name !== 'search') return result;
  return {
    params,
    result,
    source: tool.name,
  };
});

// 2) Preserve the current result and let later handlers keep processing
agent.onToolCall((result, params, tool) => {
  if (tool.name !== 'audit_only') return result;
  return skip();
});
```

# onAfterToolCall
Register a handler that runs after the final tool result has been resolved.

Use this to:
- persist audit trails
- emit metrics
- trigger external side-effects based on the final tool result

**Example**
```ts
agent.onAfterToolCall((result, params, tool, context) => {
  const audit = context.getStore('audit');
  audit?.update(prev => ({
    ...prev,
    lastTool: tool.name,
    lastParams: params,
    lastResult: result,
  }));
});
```

# onAfterStateUpdate
Register a handler that runs **after** the agent state changes (status, step count, etc.).

Use this to:
- Track lifecycle metrics (idle/generating/waiting).
- Implement watchdogs and timeouts.
- Drive external UIs or orchestrators.

**Examples**
```ts
// 1) Log status transitions
agent.onAfterStateUpdate((context) => {
  console.log('Agent status:', context.state.status);
});

// 2) Emit metrics about step count
agent.onAfterStateUpdate((context) => {
  metricsClient.gauge('agent.steps', context.state.stepCount);
});

// 3) Trigger a timeout if agent generates too long
agent.onAfterStateUpdate((context) => {
  const { stepCount, status } = context.state;
  if (status === 'generating' && stepCount > 20) {
    context.stopSync();
  }
});
```

# Unsubscribing and Composition
All `on*` helpers return an unsubscribe function. Store and call it when you no longer need the handler (for example when disposing an agent instance or test).

You can also register multiple handlers for the same event; they will run in the order they were added.

**Examples**
```ts
// 1) Unsubscribe later
const offUser = agent.onUserMessage((message) => message);
// ... later
offUser();

// 2) Compose multiple analytics handlers
const off1 = agent.onAfterStateUpdate(ctx => console.log('A', ctx.state.status));
const off2 = agent.onAfterStateUpdate(ctx => console.log('B', ctx.state.status));

// Teardown
off1();
off2();
```
